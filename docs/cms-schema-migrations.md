# CMS Schema Migrations — Workflow & Baseline Reset

How Payload/D1 migrations work in this repo, day-to-day commands, and the destructive **baseline reset** procedure we're using while still pre-prod.

Not to be confused with `content-migration.md` (which is about porting *content data* from GitHub Issues).

---

## How migrations work here

- Schema lives in `apps/cms/src/collections/*.ts` (Payload collection configs).
- Migration files live in `apps/cms/src/migrations/YYYYMMDD_HHMMSS.ts|json`, registered by `apps/cms/src/migrations/index.ts`.
- The D1 adapter (`@payloadcms/db-d1-sqlite`) reads the index and runs un-applied migrations sequentially. A `payload_migrations` tracking table records what's been applied.
- **Every migration must be generated by `pnpm migrate:create`** — the CLI knows how to handle SQLite/D1 quirks (DROP COLUMN + foreign keys, type changes, etc.). Hand-writing migrations is how you end up with scripts that work locally and blow up on D1.

The auto-apply flow: CF Workers Builds runs `build`, which is `migrate:prod && next build`. With `CLOUDFLARE_ENV=test` set as a build-time dashboard variable, `payload migrate` connects to the **remote** test D1 (because the test env's D1 binding has `"remote": true`), applies any un-applied migrations, then Next builds the worker. Deploy is atomic: if a migration fails, the worker doesn't ship. Same flow for live.

## Day-to-day

### After changing a collection schema

```bash
# 1. Generate a migration from the schema diff
pnpm migrate:create
#    Name it something descriptive when prompted (e.g. "add-gallery-caption")

# 2. Verify it runs cleanly from a fresh local DB (catches the "works with existing
#    state, fails on fresh deploy" class of bugs before push)
pnpm migrate:fresh
#    (add this alias if not present — see "Scripts reference" below)

# 3. Commit schema + migration together
git add apps/cms/src/collections apps/cms/src/migrations
git commit -m "..."
git push
#    → CF build auto-applies to remote test D1, then deploys worker
```

### If you *just* need to re-apply locally

```bash
pnpm migrate
```

Non-destructive. Runs only un-applied migrations.

## Baseline reset (destructive, pre-prod only)

Use this when the migration history is broken — e.g. a hand-written migration slipped in and can't be applied to a fresh D1, or accumulated rewrites + deletions have left a mess.

**Preconditions**: you're pre-prod (no production data to preserve), or you've exported whatever data is worth keeping.

### 1. Wipe local migrations + local DB state

```bash
# From repo root
rm -rf apps/cms/src/migrations/*.ts apps/cms/src/migrations/*.json
# `index.ts` will be regenerated in step 3 — safe to delete too
rm -f apps/cms/src/migrations/index.ts

# Local miniflare D1 state
rm -rf apps/cms/.wrangler/state
```

### 1b. Confirm what's in the remote D1 before wiping

Sanity check what's there. Three ways, increasing specificity:

```bash
cd apps/cms

# a) Total rows + storage size (fastest summary)
pnpm exec wrangler d1 info blrhikes-test --json
#   → "num_tables": 0 means fresh / empty. Non-zero means something's been applied.

# b) List user tables
pnpm exec wrangler d1 execute blrhikes-test --remote \
  --command "SELECT name FROM sqlite_master WHERE type='table' AND name NOT LIKE 'sqlite_%' ORDER BY name;"
#   → empty result = fresh DB. If you see `trails`, `users`, `payload_migrations`, etc., migrations have run.

# c) Which migrations have been applied (only works if migrations have run at least once)
pnpm exec wrangler d1 execute blrhikes-test --remote \
  --command "SELECT name, batch FROM payload_migrations ORDER BY id;"
#   → errors with "no such table" if it's a fresh DB. Otherwise lists applied migration names.
```

Run the same against `blrhikes-live` when it exists.

If (a) shows `num_tables: 0` or (b) returns nothing, the DB is empty and you can skip the delete in the next step (jump to regenerating migrations). If either returns rows, proceed with the wipe.

### 2. Wipe each remote D1 you've already deployed to

The remote already has some migrations applied + the `payload_migrations` tracking table. Cleanest is to nuke and recreate the whole DB:

```bash
cd apps/cms

# Test
pnpm exec wrangler d1 delete blrhikes-test
pnpm exec wrangler d1 create blrhikes-test --location=apac
# → copy the printed database_id into
#   apps/cms/wrangler.jsonc > env.test.d1_databases[0].database_id

# Live (only if it exists)
pnpm exec wrangler d1 delete blrhikes-live
pnpm exec wrangler d1 create blrhikes-live --location=apac
# → same, into env.live.d1_databases[0].database_id
```

If you prefer surgical: `wrangler d1 execute blrhikes-test --remote --command "DROP TABLE tablename"` for each table including `payload_migrations`. Delete + recreate is faster and avoids missing one.

### 3. Generate one clean baseline migration

```bash
# From repo root (uses the CMS's local miniflare — no CLOUDFLARE_ENV=remote needed)
pnpm migrate:create
# Name it "baseline" or "initial"
```

This diffs the (now empty) local DB against the current collection schemas → produces one migration containing every `CREATE TABLE` for the full current schema. CLI-generated, so D1-safe.

### 4. Verify end-to-end locally

```bash
rm -rf apps/cms/.wrangler/state   # fresh local DB again
pnpm migrate
```

Should apply the single baseline cleanly. If it fails here, it'll fail in CF build — fix before pushing.

### 5. Commit + push

```bash
git add apps/cms/src/migrations apps/cms/wrangler.jsonc
git commit -m "chore(cms): reset migrations to a single CLI-generated baseline"
git push
```

CF Workers Builds picks up, runs the baseline against the fresh remote test D1, deploys worker. Same happens for live on its next deploy.

### 6. Restore data (if you exported any)

```bash
pnpm exec wrangler d1 execute blrhikes-test --remote --file=backup.sql
```

## How to avoid needing a reset in the future

Ranked by value:

1. **Never hand-write migrations.** Enforced by convention in
   [`memory/feedback_no_handwritten_migrations.md`](../.claude/projects/-Users-shreshthmohan-Projects-personal-blrhikes-2/memory/feedback_no_handwritten_migrations.md)
   (user's Claude memory). Hand-written migrations bypass the CLI's SQLite/D1 quirk handling and will eventually fail on a fresh D1. The pre-commit hook (`.githooks/pre-commit`) nags when you change schema without a migration, but *can't* tell a hand-written one from a CLI-generated one — that discipline is on us.

2. **Add a `migrate:fresh` script** and run it before every push that touches a migration:

   ```json
   "migrate:fresh": "rm -rf .wrangler/state && pnpm migrate"
   ```

   Nukes local state and replays every migration from scratch. Catches the exact failure mode the baseline reset solves — before it reaches CF.

3. **CI job that replays migrations on every PR**. A ~15-line GitHub Actions workflow: `pnpm install` → `pnpm --filter cms migrate:fresh` → fail on non-zero. Stops the bug from ever reaching a deploy.

4. **Preview deploys per PR (ephemeral test D1)**. Larger infra investment. Useful later, overkill now.

Minimum practical setup: **#1 + #2**. #3 is a cheap add-on when you want CI-level enforcement.

## Scripts reference (apps/cms/package.json)

| Script | Purpose |
|--------|---------|
| `pnpm migrate` | Run un-applied migrations against local miniflare D1 (via `CLOUDFLARE_ENV=local`) |
| `pnpm migrate:create` | Diff local DB vs schema, generate a new migration file |
| `pnpm migrate:prod` | Non-interactive migrate. Used by `build`. Runs against whatever D1 the current `CLOUDFLARE_ENV` resolves to |
| `pnpm migrate:fresh` | *(add me)* wipe `.wrangler/state` + run all migrations from scratch |
| `pnpm build` | `migrate:prod && next build` — used by CF Workers Builds. Auto-migrates remote DB before shipping worker |
| `pnpm build:cf` | Build-only (no migrate). Escape hatch; normal flow shouldn't need it |
| `pnpm deploy:database` | Manual remote-D1 migrate + `PRAGMA optimize`. For out-of-band migration runs |

## Gotchas

- **Wrangler auth is per-machine**: `wrangler login` once. CF Workers Builds container has its own auth — you don't need to log in there.
- **`migrate:create` vs `migrate`**: `create` generates a new file from schema diff; `migrate` applies existing files. Don't confuse them.
- **`payload_migrations` tracks applied migrations by name**. If you rename an already-applied migration file, Payload thinks it's new and will re-apply → usually blows up on "table already exists". Never rename migration files after apply.
- **Renaming collections/fields**: `migrate:create` CLI asks interactively whether it's a rename or a drop+create. Pick carefully — wrong answer = data loss or duplicate tables.
- **`.wrangler/state/` is gitignored**. It's your local miniflare SQLite. Wiping it is always safe.
